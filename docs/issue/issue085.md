# Issue Template (Duplicate & Rename)

> Rename this file to the next available `issue0XX.md` (starting from `issue085.md`) before editing.

## 1. Summary
- What problem are we solving?
- Why is it important now?

## 2. Background & Context
- Links to previous issues, documents, or user feedback.
- Current behaviour or constraints to keep in mind.

## 3. Goals & Non-Goals
- **Goals**: Facts that must be true after this work ships.
- **Non-Goals**: Out-of-scope items to avoid scope creep.

## 4. Proposed Approach
- High-level solution outline (architecture, data flow, affected modules).
- Open questions or decisions waiting for confirmation.

## 5. Implementation Checklist
- [ ] Step-by-step technical tasks referencing files or directories.
- [ ] Tests or tooling updates required.
- [ ] Documentation updates (policy/dev/feature) to schedule.

## 6. Validation Plan
- Automated tests to run (unit, integration, E2E).
- Manual checks or QA scenarios.
- Metrics or logs to observe after release.

## 7. Rollout & Follow-up
- Deployment notes (feature flags, environment changes).
- Post-release monitoring or follow-up issues.

## 8. Status Log
| Date | Update | Owner |
| --- | --- | --- |
| YYYY-MM-DD | Initial draft | Meti |

Keep this document up to date throughout implementation; it is the authoritative record for the issue.
