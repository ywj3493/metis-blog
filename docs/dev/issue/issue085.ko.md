# Issue 템플릿 (복제 및 이름 변경)

> 편집하기 전에 이 파일을 사용 가능한 다음 `issue0XX.md`로 이름을 변경하세요 (`issue085.md`부터 시작).

## 1. 요약
- 어떤 문제를 해결하고 있나요?
- 왜 지금 중요한가요?

## 2. 배경 및 컨텍스트
- 이전 issue, 문서 또는 사용자 피드백에 대한 링크.
- 염두에 두어야 할 현재 동작 또는 제약 사항.

## 3. 목표 및 비목표
- **목표**: 이 작업이 완료된 후 반드시 참이어야 하는 사실들.
- **비목표**: 범위 확장을 피하기 위해 범위에서 제외되는 항목들.

## 4. 제안된 접근 방법
- 고수준 솔루션 개요 (아키텍처, 데이터 흐름, 영향을 받는 모듈).
- 확인을 기다리는 미해결 질문 또는 결정 사항.

## 5. 구현 체크리스트
- [ ] 파일 또는 디렉토리를 참조하는 단계별 기술 작업.
- [ ] 필요한 테스트 또는 도구 업데이트.
- [ ] 예약할 문서 업데이트 (policy/dev/feature).

## 6. 검증 계획
- 실행할 자동화된 테스트 (단위, 통합, E2E).
- 수동 확인 또는 QA 시나리오.
- 릴리스 후 관찰할 메트릭 또는 로그.

## 7. 출시 및 후속 조치
- 배포 노트 (기능 플래그, 환경 변경).
- 릴리스 후 모니터링 또는 후속 issue.

## 8. 상태 로그
| 날짜 | 업데이트 | 담당자 |
| --- | --- | --- |
| YYYY-MM-DD | 초기 초안 | Meti |

구현 전반에 걸쳐 이 문서를 최신 상태로 유지하세요. 이는 issue의 권위 있는 기록입니다.
